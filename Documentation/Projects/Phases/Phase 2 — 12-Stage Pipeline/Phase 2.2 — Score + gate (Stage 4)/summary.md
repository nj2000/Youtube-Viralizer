## Deviations from `task.md`

1. **`stage-handlers.ts` barrel added (not in task.md).** Phase 2.1 left a latent gap: the orchestrator's `rerun-from` route never imported `lib/services/competitor.ts`, so Stage 3 handler registration could be skipped under cold-start cache paths. Phase 2.2 ships a side-effect barrel imported by `lib/services/pipeline.ts` to fix this for both stages and every future one.

2. **`markGateFailed` signature extended.** Task.md's Step 1 listed `score_data` persistence implicitly; the existing helper only flipped status. Without persisting on gate fail, the model's reframes (the whole point of Phase 2.2) would be lost. The signature change is backward-compatible (`scoreData?: Json`).

3. **`reframe_applications` schema fuller than task.md.** Task.md spec'd `{id, run_id, reframe_index, applied_at}`; the migration also stores `user_id`, `original_idea_text`, `revised_idea_text`, `expected_score_lift` (per spec §3) so Feature #17 (Phase 3 calibration) has the audit context it needs without back-filling.

4. **Apply-reframe wipes all 10 stage-data columns, not just competitor + score.** Task.md says "wipe competitor_data + score_data". But if a user overrode the gate, ran some downstream stages, then applied a reframe — the downstream data would still reference the OLD `idea_text`. Cleaner to wipe everything in one statement; the orchestrator overwrites as stages re-run.

5. **`RunView.tsx` is 245 lines** (Q-2 component cap is 200). Same trade-off as Phase 2.1's service file: it's the top-level page composition. Splitting `StatusPill` out would fragment the obvious code-locality of "what's in the page header". Documented as conscious trade-off.

6. **Migration NOT applied to the live DB.** Per the Phase 1.2 IPv4-pooler gotcha (CLAUDE.md EXT-4), running `pnpm db:push` from this session requires re-linking the Supabase project. The migration file is checked in; the user should run `supabase link --project-ref <ref>` then `pnpm db:push` to apply, then `pnpm db:types` to regenerate `lib/db/types.ts` (which should match the hand-edited version in this commit).

- **Hybrid scoring with empirical corpus** — Feature #14, Phase 3
- **Per-niche calibration multipliers** — Feature #17, Phase 3
- **Score caching across calls** — only the system-prompt cache is shared; each score call hits Anthropic
- **Score history / past-score audit log** — Phase 2 polish; current schema persists only the latest score
- **Reframe-quality validation** — verifying a reframe's predicted lift actually materializes; Phase 3 calibration loop

Other follow-ups specific to this phase:

- **Apply migration to remote DB:** `supabase link --project-ref <ref>` then `pnpm db:push`, then `pnpm db:types` to regenerate `lib/db/types.ts`. Hand-edited types should match the regen byte-for-byte.
- **`lib/services/pipeline.ts` `isGateFailed` checks `passed === false`** — this is correct for our new shape since `passed` is computed in TS alongside `finalScore`. If a future score handler emits `passed: true` with `finalScore < 92` (which `computeFinalScore` + the assignment in `score.ts` prevent today), the score-fallback in `isGateFailed` correctly catches it.
- **Stage 5 (titles) is the next dependent stage.** It reads `score_data` and is unblocked the moment Phase 2.3 ships.

# PRD — Calibration Loop

## Feature Name
Calibration Loop

## Overview
Tracks which scored ideas the user actually published, fetches their real performance after publish, compares to the predicted score and forecast, and recalibrates the scoring engine per niche. Closes the loop between predicted virality and observed virality.

**Problem solved:** Without calibration, scoring drifts. The system becomes overconfident or underconfident over time and there's no signal to correct it. The calibration loop turns predictions into measurements that improve future predictions.

## User Stories
- As a creator, I want the system to remember which ideas I actually published, so it learns my channel's real performance.
- As a creator, I want to see how accurate the predictions have been, so I can trust the score over time.
- As a product owner, I want per-niche calibration data, so scoring quality improves measurably.
- As a creator, I want the system to flag when it was wrong, so I learn what the predictor missed.

## Functional Requirements
- "Mark as published" action on a run → user supplies the YouTube video URL
- System polls YouTube for that video's metrics every 24h for 30 days post-publish (subject to quota)
- Persists `published_outcome` table: `runId`, `videoId`, daily metrics snapshot (views, AVD, CTR if accessible)
- After 30 days, run a calibration analysis:
  - Predicted vs. actual 30-day view count (from Feature #16)
  - Predicted vs. actual AVD (from Feature #15)
  - Predicted gate pass/fail vs. actual outlier status (5×+ channel median)
- Aggregate calibration data per niche → adjustment factors fed back into Feature #14 hybrid scoring
- Surface per-channel calibration metrics in a "Performance" view
- "Predicted accuracy: ±N%" badge appears on each new score once the user has 5+ calibrated runs

## User Interface

### Screens
- **Run view enhancement**: each completed run gains a "Mark as published" CTA
- **`/performance`**: new page summarizing prediction accuracy across the user's published runs
  - List of published runs with predicted vs. actual columns
  - Aggregate metrics: average prediction error, calibration trend over time
  - Per-niche breakdown if user has multiple channels

### Key interactions
- "Mark as published" prompts for the YouTube URL, validates, and starts the polling job
- Performance page is read-only

## States to Handle

### Happy path
User publishes → marks published → system polls → 30-day window completes → calibration computed → score adjustments propagate.

### Error states
- Marked video URL invalid or doesn't belong to the user's channel → reject with explanation
- YouTube quota constrains polling → reduce polling cadence to 48h instead of 24h; flag in admin
- Video deleted or made private mid-window → freeze last known metrics; flag

### Empty states
- User has no published runs yet → performance page shows "Publish a kit and mark it published to see calibration data"
- Window not yet complete (less than 30 days) → show partial data + "Final calibration in N days"

### Loading states
- Polling status indicator on the performance page

## Edge Cases
- User publishes a video that wasn't generated by our app → cannot calibrate; ignore
- User publishes the same idea twice (rare) → calibrate against the run that was actually filmed
- Video performance is heavily influenced by external factor (Reddit traffic spike, sponsor boost) → calibration may be noisy; aggregate over many runs to smooth
- User has hidden subscriber count or analytics → calibrate on public metrics only
- User's channel velocity changes mid-window (e.g., stops uploading) → flag the run as low-confidence calibration
- 30-day window is short for slow-burn topics → run a 90-day calibration as a secondary signal

## Out of Scope
- Real-time prediction adjustments (calibration is a periodic process, not per-call)
- Surfacing per-creator calibration as a competitive differentiator
- Pulling YouTube Analytics via OAuth (Phase 3+)
- Calibrating against multiple independent runs of the same idea
- Notifying the user when their channel calibration drift exceeds a threshold
